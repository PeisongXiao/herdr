---
title: How to work with Herdr
description: Run Herdr locally, inside SSH, or through remote attach.
---

Run Herdr where the work lives. Attach from wherever you are.

Herdr is a background session server plus one or more terminal clients. Panes keep running in the server. Clients attach, detach, and render the session.

## Local work

Start Herdr from the project directory:

```bash
herdr
```

Herdr starts or attaches to your local background session automatically. You do not manage sockets. Run shells, servers, tests, and agents normally inside panes.

Detach the client with `ctrl+b q`. Your panes keep running.

Reattach later:

```bash
herdr
```

If you want to end the session and stop its panes, stop the server:

```bash
herdr server stop
```

## SSH from inside Herdr

On Linux and macOS, the shortest path from local work to remote work is:

```bash
herdr
# run this inside a Herdr pane
ssh you@server
```

OpenSSH handles host keys, passwords, passphrases, and MFA normally. After authentication, Herdr automatically finds or starts the remote Herdr daemon, connects the two servers, and turns the current pane into a shell backed by the remote default Herdr session. You do not start a second TUI or manage a socket.

The remote machine must already have a compatible Herdr that advertises peer federation and exclusive remote-pane presentation. Herdr does not install or upgrade it on this path. If the binary is missing, the protocol or capabilities are incompatible, forwarding is blocked, or managed setup otherwise fails, Herdr prints a warning and reruns the original command as ordinary SSH.

Managed SSH moves pane presentation rather than making a second copy. The pane appears in a workspace on exactly one machine, and the owner is responsible for its lifetime. Closing the owner pane, stopping its Herdr server, or losing the SSH connection destroys the delegated remote pane and its process. Run `herdr remote-handoff` from inside that remote pane to move the pane under your cursor into a new workspace on its host before the connection closes. See [Persistence and remote access](/docs/persistence-remote/) for nested SSH and explicit takeover behavior.

## Run Herdr after normal SSH

SSH to the machine that has the code and credentials, then run Herdr there:

```bash
ssh you@server
herdr
```

This works like a terminal multiplexer. Your shell is remote. The Herdr server is remote. The agents and panes run on the remote machine. Detach with `ctrl+b q`, disconnect, then SSH back and run `herdr` again. Start the SSH command outside Herdr, or use `HERDR_SSH_BYPASS=1 ssh you@server` from a Herdr pane, when you specifically want this unmanaged path.

Use this path when you already live inside an SSH shell, when you are on a phone or tablet SSH client, or when you want the simplest possible setup.

## Work from your phone

You do not need a Herdr mobile app or a web dashboard. Install any SSH client on your phone, connect to the machine where your agents run, and start Herdr there:

```bash
ssh you@server
herdr
```

The same persistent Herdr session opens in your phone terminal. The TUI adapts to narrow screens, so you can inspect agents, switch workspaces, and check panes without leaving SSH.

On iPhone, apps like [moshi](https://getmoshi.app/) work well.

## Remote work from your local terminal

Attach through SSH without opening a shell first:

```bash
herdr --remote workbox
herdr --remote ssh://you@server:2222
```

Your local Herdr acts as a thin client. It connects over SSH, starts or attaches to the remote Herdr server, and streams the UI back to your local terminal.

Use this path when you want the remote session to feel local. The client runs on your machine, so local desktop features such as image clipboard paste can be bridged to the remote server. If you SSH first and run `herdr` on the server, Herdr runs entirely on that server and cannot read your local desktop clipboard.

For repeat targets, put the host in your SSH config:

```text
Host workbox
  HostName server.example.com
  User you
  Port 2222
```

Then attach with:

```bash
herdr --remote workbox
```

## Which path to use

Use `herdr`, then `ssh you@server` inside a pane, for the direct Unix workflow where one local workspace presents an exclusively owned remote shell. Use `herdr agent start --ssh ... --ssh-session NAME` when an explicit remote agent should use a named session. Use ordinary SSH from outside Herdr, then run `herdr`, when you want durable tmux-style remote work or when you are using a phone SSH client. Use `herdr --remote <host>` when you want the complete remote Herdr UI as a local thin client, automatic remote bootstrap, and local clipboard image paste bridging.

For remote bootstrap details, named remote sessions, custom binaries, direct terminal attach, and `--no-session`, see [Persistence and remote access](/docs/persistence-remote/).
